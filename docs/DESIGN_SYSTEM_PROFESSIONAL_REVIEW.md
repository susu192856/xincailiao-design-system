# 设计系统专业度审计与补齐说明

生成日期：2026-06-11

本文件站在资深 UI / 设计系统设计师视角，判断当前新材道设计规范是否已经具备对设计师、前端和 Figma 组件库都可用的专业度，并补齐后续页面优化必须遵守的规则。它不是视觉稿替代品，而是后续逐页修正的验收标准。

## 结论

当前规范已经具备“设计规范网站”的基础专业度：有首页定位、基础 Token、组件菜单、React 示例组件、Vue 组件起点、Figma Token 和组件清单。它已经不是单纯展示页，而是一个初级规范工具。

但它还没有达到成熟设计系统的完整程度。主要缺口不在“有没有页面”，而在“规则是否足够可执行”：部分组件页面仍偏展示，缺少场景边界、尺寸数值、状态定义、组合规则、Figma 组件状态说明、前端 API 对齐说明和真实业务案例。

## 适用性判断

### B 端 / 后台系统

方向基本适合。黑白灰为主、品牌红克制使用、产品蓝用于功能焦点，能够支撑数据空间、材库、AI 工具和后台管理场景。

需要继续补强：

- 高密度表格、筛选、分页、批量操作和权限状态的组合规范。
- 宽页面表单布局和字段最大宽度。
- 树、穿梭框、折叠面板、弹窗、抽屉等后台高频组件的交互状态说明。

### 官网 / 门户平台

方向可以共用，但不能简单复用后台密度。官网和门户应使用更大的留白、更强的叙事区块、更少的表格密度和更明确的 CTA 层级。

需要继续补强：

- 官网一级、二级、三级页面骨架示例。
- 品牌红作为关键转化色的使用边界。
- 门户场景的导航、面包屑、Banner、详情页左右分栏规范。

### 共同使用的前提

同一套 Token 可以共用，但组件在不同业务层必须允许密度、尺寸和内容结构差异：

- 官网 / 门户：叙事优先，内容宽度以 `1400px` 居中为主。
- B 端 / 后台：效率优先，页面宽度以 `1440px` 设计画布和自适应工作区为主。
- 组件层：同一组件 API 保持一致，视觉尺寸和密度按场景切换。

## 展示页还是规范工具

当前是“规范工具雏形”，不是纯展示页。理由：

- `src/styles/tokens.css` 是 Token 源码。
- `figma/tokens.json` 可导入 Figma Tokens / Tokens Studio。
- `figma/components.manifest.json` 已经描述组件命名、路由、源码和状态。
- `src/components/ui` 有 React 规范站组件。
- `packages/vue-ui` 有 Vue 组件源码起点。
- `/components/*` 页面已指向真实组件页面。

还未成为成熟规范工具的原因：

- Figma Components 尚未自动生成或完全同步。
- 组件页面的规则说明还不均衡，部分页面缺少真实业务组合案例。
- 缺少可访问性、键盘操作、响应式、错误处理、长内容处理和跨组件一致性的统一验收标准。
- 缺少组件视觉回归和交互测试。

## 全局补齐规则

### 色彩规则

色彩角色和色彩语义模型不冲突，但必须分层解释：

- 色彩角色回答“这个颜色在业务里承担什么责任”。
- 色彩语义回答“这个颜色在状态里表达什么含义”。
- 品牌红用于品牌识别、关键转化、关键节点和重要提示，不用于后台常规操作。
- 产品蓝用于后台功能操作、焦点、链接和可交互状态，不等于次按钮。
- 错误、警告、成功、信息必须使用语义色，不用品牌红替代错误色。
- 数据色用于图表序列，不应默认使用品牌红。品牌红只在图表中表达关键阈值、异常结果或业务重点。

数据色 12 色可作为初始图表色板，但使用原则需要更严格：

- 1 到 6 条序列优先使用 `data-01` 到 `data-06`。
- 7 到 12 条序列可启用 `data-07` 到 `data-12`，但需要配合线型、点型、透明度或图例分组，不仅依赖颜色区分。
- 面积图、饼图不建议同时使用 12 色，超过 6 类应合并为“其他”。
- 风险、异常、阈值线优先使用 `error-text` 或 `warning-text`，不要占用普通数据序列。
- 图表示例应做成可复用组件，至少覆盖折线图、柱状图、堆叠柱、环图、面积图、散点/关系图、阈值线示例。

### 字体规则

字体页面和全站应以 Token 为准：

- 一级页面标题：非首页使用 H2 级别；首页可使用更大的展示标题。
- 二级标题：使用 H4 / H5 级别，保持与首页章节标题一致。
- 正文：优先使用 Body/M，重要说明可用 Body/L。
- 辅助说明：使用 Body/S 或 Caption。
- 按钮文字：常规字重，除非需要强调主 CTA，不应默认加粗。

推荐行长以 `700px` 为正文阅读上限。`760px` 可以作为文档内容区或说明段落容器宽度，但不应作为长正文的推荐行长。最终口径：

- 正文舒适阅读：`640px-700px`。
- 页面说明容器：最多 `760px`。
- 表格、代码、栅格示例不受正文行长限制。

### 圆角规则

组件默认圆角应以 `radius-sm = 2px` 为主：

- 按钮、输入框、选择器、标签、分页、开关：2px。
- 表格、卡片、折叠面板、描述列表：2px 到 4px。
- 弹窗、抽屉：4px 到 8px，后台优先 4px。
- 官网展示型大卡片可到 8px，但不能影响后台组件的一致性。

页面卡片和弹窗如果超过 8px，需要明确是营销展示场景，否则应收敛。

### 阴影和操作顺序

主次按钮同时出现时，默认规则如下：

- 水平按钮组：次要 / 取消在左，主要 / 确认在右。
- 弹窗底部：取消在左，确认在右；危险确认仍在右侧，但必须二次确认。
- 工具栏：主按钮可放在最右，筛选和重置靠近筛选区域。
- 移动端：主要按钮可置顶或置底，次要按钮在其下方。

通知和提示图标必须跟随语义：

- 信息：`info-text` 或产品蓝。
- 成功：`success-text`。
- 警告：`warning-text`。
- 错误：`error-text`。
- 重要通知不应默认使用黑色图标，除非它只是普通内容卡片而非状态提示。

### 表单和输入规则

输入框必须同时说明“带名称”和“不带名称”的适用场景：

- 带名称：表单录入、配置项、需要解释字段含义的场景。
- 不带名称：搜索、筛选、工具栏、上下文已明确的轻量操作。
- 上下结构：适合常规表单、移动端、字段解释较多的场景。
- 左右结构：适合宽后台页面、字段多且标签短的配置型表单。

尺寸口径应与输入框、选择器、按钮保持一致：

- 小尺寸：28px，后台紧凑筛选和表格内编辑。
- 中尺寸：32px，后台默认。
- 大尺寸：36px 或 40px，用于官网、门户和重点操作区。
- 表单项标签字号通常不随输入框尺寸变化，默认 13px / 14px，字重常规或中等；只在重点区块标题中增强。

禁用和只读必须区分：

- 禁用：不可聚焦、不可复制或不可提交，表示当前条件下不可用。
- 只读：可聚焦、可复制、可查看，但不可编辑，常用于详情页和审批页。

宽页面表单规则：

- 表单区宽度超过 1000px 时，优先使用 2 列或 3 列栅格。
- 单个输入框推荐最大宽度为 360px 到 480px；长文本、地址、说明类字段可占满整行。
- 复杂数据配置页应提供分组标题、分割线和固定底部操作区。

### 表格规则

表格必须补齐以下规范：

- 文本列左对齐，数字列右对齐，状态列居中或左对齐需统一，操作列右对齐。
- 表头字号、颜色和分页页面里的表格必须一致。
- 筛选区字段需要名称；当上下文清晰时可用 placeholder，但文档中必须说明。
- 筛选与重置同时出现时，重置在左，查询在右；空间不足时查询保留，重置变为文字按钮或收进更多操作。
- 表格长内容需要说明：省略、换行、Tooltip、横向滚动和固定列。
- 分割栏表格适合材料成分、参数对比和多指标矩阵，应作为独立表格类型补充。
- 行状态不要同时堆叠背景色、状态标签、操作列色彩和勾选色；优先使用单一信号。

复选框必须跨表格、树、穿梭框保持一致尺寸和状态。

### 标签与状态规则

标签和状态不是同一件事：

- 分类标签：表达类型、归属、业务分类，可多色但弱视觉。
- 状态标签：表达流程状态、风险状态、启停状态，颜色必须有语义。
- 强提醒标签：只用于高风险、异常、需关注状态，不应在普通分类里使用。

标签超过 6 到 8 个颜色时，必须建立色板规则：

- 普通分类优先使用中性色或浅色背景。
- 业务类型可使用固定色，但不要和语义状态色冲突。
- 高风险标签不能使用过重背景，除非在告警面板中。

### 菜单和标签页规则

菜单需要区分三个层级：

- 一级菜单：页面级或系统级导航，承载主模块切换。
- 二级菜单 / Tabs：当前页面内容分区切换。
- 三级切换 / Segmented：图表时间维度、局部视图或筛选维度切换。

顶部菜单不一定需要图标。若文字空间不足，优先减少图标或改为下拉，不应压缩文字。

菜单标签页间距不宜过密。后台页面空间足够时，应提高标签可点击区域，并确保当前态不过度抢主按钮层级。

### 弹窗、抽屉和浮层规则

弹窗页面必须补齐：

- 普通弹窗、确认弹窗、危险确认弹窗。
- 抽屉：右侧详情、筛选、配置、审批流查看。
- Figma 调用方式：展示静态面板、打开态、关闭态、遮罩态和底部操作区，不依赖点击交互才能查看。
- 操作顺序：取消左、确认右；关闭 icon 在右上角；危险操作必须有二次确认。

Tooltip、Popover 和 Drawer 已补齐为独立组件，后续仍需补 Chart / 数据可视化：

- Tooltip：短文本解释，支持上、下、左、右。
- Popover：轻量信息面板，可承载说明、快捷操作、字段解释。
- Drawer：右侧详情、筛选、配置、审批流查看。

### 图片、头像、徽标和反馈规则

图片需要补齐比例和占位：

- 常用比例：1:1、4:3、16:9、3:2、宽横幅。
- 占位不应全用图标；应区分加载、空图、失败、无权限。

头像需要补齐：

- 图片头像、姓名缩写头像、匿名头像、系统头像、组织头像。
- 无真实姓名时使用账号、手机号尾号、系统默认或组织名首字。

徽标数需要明确位置和语义：

- 右上角用于数量提醒。
- 右下角用于在线状态或身份标记。
- 同一头像不应同时出现多个无解释状态点。

Toast / Alert 反馈：

- Toast 不宜过重，优先用语义图标 + 文案。
- 如果已经有语义色条，不应再叠加同等强度色点。
- 需要常驻提示、可关闭提示和操作型提示。

## 后续优先级

### P0：影响三端可用性的基础规则

1. 颜色角色、数据色、字体行长、圆角、按钮顺序、标签语义。
2. 输入框、选择器、表单、表格的尺寸和组合规则。
3. Figma 组件状态：默认、悬停、聚焦、禁用、只读、错误、加载、展开。

### P1：后台高频组件修正

1. 表格、分页、筛选区、树、穿梭框、折叠面板。
2. 弹窗、抽屉、Toast、Tooltip、Popover。
3. 菜单、Tabs、多层导航。

### P2：视觉质感与业务场景增强

1. 卡片、描述列表、图片、头像、徽标数。
2. 官网 / 门户和 B 端页面骨架示例。
3. 图表组件和数据可视化场景。

## 三端验收标准

### 网页查看

- 每个基础规范页和组件页必须有明确标题、用途、尺寸、状态、使用建议和反例。
- 复杂组件必须提供真实业务场景，不只展示单个样式。
- 页面内容要能被设计师快速理解，也能被开发直接查 API。

### 前端使用

- 页面示例优先引用 `src/components/ui` 的真实组件。
- React 和 Vue API 命名保持一致。
- 尺寸、颜色、状态来自 Token 或组件 props，不写孤立临时样式。
- 组件状态要覆盖 disabled、readonly、error、loading、empty、selected、checked 等真实状态。

### Figma 调用

- Token 从 `figma/tokens.json` 导入。
- 组件属性参考 `figma/components.manifest.json`。
- 每个组件在 Figma 中至少应有默认态、禁用态、错误态、聚焦/选中态、尺寸变体。
- 弹窗、抽屉、下拉、Tooltip、Popover 这类浮层必须以静态打开态组件呈现，不能只依赖原型点击。

## 专业度评级

当前评级：B-。

已具备：

- 设计规范网站基础形态。
- Token 和组件交付链路。
- 面向 Figma、前端和网页的初始结构。

未达到 A 级的原因：

- 规则说明仍不够细。
- 组件视觉质量不均衡。
- 多个复杂组件缺少真实业务场景。
- Figma Components 尚未完成。
- 缺少交互、可访问性、响应式和视觉回归验证。

达到 A 级需要：

- 完成 P0 和 P1 优先级修正。
- 所有组件页补齐尺寸、状态、组合、场景和 Figma 调用说明。
- 建立 Chart / 数据可视化等缺失组件，并继续完善 Tooltip、Popover、Drawer 的可访问性实现。
- 组件在 Figma、React、Vue、网页四处命名和状态完全一致。
