# 规则治理

## 权威顺序

设计负责人确认 > Token > 组件 manifest > 已验收代码 > 已验收 Figma > 重复案例。

## 新增或修改规则

1. 确认现有规则是否已经覆盖。
2. 标记来源和影响组件。
3. 判断是 Token、硬规则、判断规则还是案例。
4. 更新最窄的 reference 或结构化资产。
5. 同步网页、React、Vue、Figma 和文档。
6. 更新 changelog 和 eval。

## 冲突

真实冲突不得静默选择。记录冲突，并由设计负责人裁决；迁移期兼容别名必须标记 deprecated。

## 案例门槛

单个 bug 或审美反馈只进入 case study。只有根因明确、来源可信且能够推广时才升级为规则。
